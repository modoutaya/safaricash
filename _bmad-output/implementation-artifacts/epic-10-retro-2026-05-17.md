## What went well

- **The 3-layer adversarial code review earned its keep.** Across the 5 stories it surfaced 4–8 patches each (idempotency guards, `networkMode`, race conditions, test-coverage gaps) — every one Low/Medium, every one fixed before merge. No Critical/High defect reached `done`.
- **The psql migration smoke-test discipline** (`begin`/`rollback`) repeatedly caught Postgres-side errors that TypeScript cannot see — load-bearing on every migration.
- **Deviations were documented honestly** in the Dev Notes — the non-interactive per-row dispute icon (10.3, an a11y constraint), the decision to hash server-side identifiers rather than the plaintext phone (10.4), the `anonymised_at` gate (10.5).
- **Speed** *(Project Lead)* — the create-story → dev-story → code-review loop moved fast; the whole epic shipped in one working session.

## What didn't go well / friction

- **CI failing intermittently** *(Project Lead — the top frustration)*. Root causes seen across the epic:
  - the `audit_emit` stale-rebase trap (see below) — a real bug that reached a PR;
  - the `node_modules` / `deno.lock` pollution from Deno test runs breaking the subsequent vitest run;
  - the pre-existing local `sms-inbound` / `sms-worker` Termii-secret test gap producing ~8–11 "unrelated" `test:edge` failures on *every* story — noise that erodes the signal of a real failure.
- **The `audit_emit` rebase trap.** Every story touching `audit_emit` (10.1, 10.3, 10.4) had to `CREATE OR REPLACE` from the *latest* definition. Story 10.1 shipped a bug from rebasing on a stale baseline — caught only by the full Playwright suite, not earlier. It became a memory entry and a code-review check, but it remained a manual discipline.
- **Every code review needed 4–8 patches.** `dev-story` produced solid implementations but consistently left the same finishing gap: missing idempotency guards on status-changing UPDATEs, missing `networkMode: 'always'` on user mutations, incomplete test assertions.
- **Sprint-status bookkeeping drift.** Stories 6.1–6.5 and Epics 1–5 carried stale `review` / `in-progress` statuses long after merge — surfaced and corrected only at the very end.

## Recurring patterns

- Shared Postgres functions (`audit_emit`, `format_sms_body`, `set_member_sms_opt_out`, `get_receipt_payload`, `enqueue_sms_on_transaction`) are repeatedly `CREATE OR REPLACE`d across stories — the rebase-on-the-latest-definition discipline is the project's single most error-prone manual step.
- The code-review patch set is predictable story-to-story — the same handful of categories.

## Action items

1. **Provision the local `test:edge` secrets so the suite runs clean.** Set `TERMII_INBOUND_SECRET` (and any Termii/WhatsApp test secrets) in the local edge runtime so `sms-inbound` / `sms-worker` pass locally. Removes the recurring "~8 failures, unrelated" noise — a failure then *means* something. (Directly targets the CI-flakiness frustration.)
2. **Eliminate the `node_modules` / `deno.lock` pollution.** Investigate a Deno test setup that does not repopulate `node_modules/.deno` (a dedicated dir, or dropping `--node-modules-dir=auto`) so `test:edge` and `npm run test` can run back-to-back without an `npm ci` recovery.
3. **Turn the `audit_emit` rebase discipline into a guardrail.** Add a contract test (or CI check) that fails if a migration's `audit_emit` redefinition drops a prior CASE branch — converts the most dangerous manual step into an automated gate.
4. **Pre-empt the common code-review patches in `dev-story`.** A short pre-finish checklist: an idempotency guard on every status-changing UPDATE, `networkMode: 'always'` on every user-initiated mutation, assert every column a test `select`s. Should shrink the 4–8 patches/story.
5. **Flip story status to `done` at merge time** — keep `sprint-status.yaml` honest as you go, so no drift like the 6.1–6.5 / Epics 1–5 cleanup recurs.
